# ANSVK UI Development Workflow: Data-Driven Theming

## Overview
This document outlines the workflow for developing the ANSVK UI system. It utilizes a database-driven approach for rapid iteration during the development phase, followed by a build-time code generation step to hardcode UI values for high-performance, offline-first production.

## Canonical Flutter application stack (accepted)
* **State:** [Riverpod](https://pub.dev/packages/flutter_riverpod)
* **Routing / navigation:** [go_router](https://pub.dev/packages/go_router)

Implementation rules, folder layout, symlink preference for UI docs, and AI guardrails live in **`9 Flutter UI conventions.md`**. Component APIs in **`8 Component contracts.md`**. ERP page layouts in **`10 Page patterns.md`**. Figma handoff in **`11 Figma to implementation.md`**.

## Phase 1: Component Standardization (The Lock)
Define and lock the essential components required for the Flutter Web application.

### Baseline Component List
* **Layout & Navigation:** AppScaffold, AppSidebar (Menu), AppHeader
* **Containers:** AppCard, AppModal/Dialog
* **Typography:** Text styles (H1, H2, H3, Body, Caption)
* **Data Entry (Forms):** AppTextField, AppDropdown, AppDatePicker, AppCheckbox
* **Actions:** AppButton (Primary, Secondary, Destructive, Text-only)
* **Data Display:** AppTable (Core for inventory/HR), AppBadge (Status flags)
* **Feedback & System:** AppSnackbar (Toasts), AppSyncBanner (Offline/Sync status)

## Phase 2: Visual Design (Figma)
* Design only the locked components in Figma.
* Enforce the "Clean Enterprise + Data-first" philosophy.
* Establish strict design tokens (spacing, radius, colors) within the Figma file.

## Phase 3: Tokenization & Database Architecture
Extract design variables from Figma and store them in the development database.

### Example Table: `color_tokens`
| token_name | hex_value | description |
| :--- | :--- | :--- |
| primary_color | #0F172A | Deep professional blue |
| accent_color | #6366F1 | Active states and buttons |
| danger_color | #EF4444 | Destructive actions / Offline status |

### Example Table: `component_variables`
| component | property | token_reference |
| :--- | :--- | :--- |
| AppCard | border_radius | radius_lg (e.g., 16.0) |
| AppButton | padding_horizontal | spacing_lg (e.g., 24.0) |

## Phase 4: Development Phase (Dynamic UI)
* The Flutter app queries the UI database tables during development.
* Components render dynamically based on database values.
* **Advantage:** Instantly preview UI changes across the entire app by updating database records, without needing to recompile the Flutter code.

## Phase 5: Production Deployment (Build-Time Code Generation)
* Execute a pre-build script to pull the final, approved UI variables from the database.
* Automatically generate a static Dart file (e.g., `lib/ui/design_tokens.dart`).
* **Advantage:** The production application reads hardcoded values. This eliminates UI-related database queries, ensuring zero performance hit and seamless offline functionality.
